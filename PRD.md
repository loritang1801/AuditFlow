# AuditFlow PRD

- 版本：v0.1
- 日期：2026-03-16
- 产品定位：面向中小企业的 `SOC 2` 证据管理 Agent

## 1. 产品摘要

AuditFlow 是一个以 `SOC 2` 审计准备为切入点的 AI Native 合规工作流产品，帮助中小企业将分散在文档、截图、工单、知识库和资产清单中的审计证据自动归档、映射到控制项、识别缺口，并生成可复核的审计说明与导出包。

v1 目标不是替代审计师，而是显著降低企业准备审计证据的时间成本、降低遗漏率，并把“证据是否充分、映射是否合理”这件事从纯手工工作变成“AI 初稿 + 人工复核”的协作流程。

## 2. 背景与问题定义

中小企业在准备 `SOC 2 Type I / Type II` 审计时，通常面临以下问题：

1. 证据分散在 `Confluence`、`Jira`、共享盘、云平台截图、制度文档和导出表格中，缺乏统一归档。
2. 控制项语言抽象，非专业团队很难快速判断某份证据应该映射到哪些控制项。
3. 证据新鲜度、覆盖度和冲突情况很难持续追踪，导致临近审计时集中补材料。
4. 审计说明、整改建议和补证清单大量依赖人工整理，重复性高。
5. 每轮审计都会积累隐性经验，但这些经验没有沉淀成组织级记忆，下一次仍从头开始。

## 3. 目标与成功标准

### 3.1 业务目标

1. 将审计准备周期缩短 `40%` 以上。
2. 让单个合规 owner 可以管理更多控制项和证据，而不显著增加手工整理成本。
3. 把企业内部对审计要求的理解沉淀为可复用的知识资产。

### 3.2 产品目标

1. 支持从多来源导入证据，并自动标准化为可搜索、可引用的证据对象。
2. 自动将证据映射到控制项，并给出可追溯的理由与引用位置。
3. 自动识别证据缺口、过期证据和冲突证据，并生成待办清单。
4. 提供人工复核工作台，允许 reviewer 对映射结果进行确认、修订、驳回和反馈。
5. 输出审计说明草稿和证据包，支持后续发送给外部审计师或内部负责人。

### 3.3 北极星指标

1. `Auto-mapped control coverage`：已导入证据中可自动映射控制项的比例
2. `Reviewer acceptance rate`：AI 映射结果被 reviewer 直接接受的比例
3. `Time-to-audit-pack`：从导入原始材料到生成首版审计包的时间
4. `Gap closure cycle time`：缺口被识别到补齐的平均时间

### 3.4 v1 验收阈值

1. 对高频控制项，证据映射 precision 达到 `>= 80%`
2. reviewer 对 AI 生成说明的接受率达到 `>= 70%`
3. 首版审计包生成时间比纯手工流程降低 `>= 50%`
4. 所有 AI 结论都必须带引用回链，不允许无依据断言

## 4. 非目标

v1 不做以下能力：

1. 不替代外部审计师作出最终合规结论
2. 不做通用合同审查或通用法务助手
3. 不覆盖所有合规框架，只聚焦 `SOC 2`
4. 不自动修改源系统中的证据或审批结果
5. 不做复杂企业级多层审批、跨法人实体治理和大规模权限模型

## 5. 目标用户

### 5.1 核心用户

1. `Security / IT Owner`
   负责制度落地、系统配置、证据准备，最需要降低手工整理成本。
2. `CTO / 技术负责人`
   对审计进度负责，希望快速知道当前缺口和风险项。
3. `Compliance Manager / GRC Owner`
   需要控制项覆盖、补证推进和审计说明的一致性。
4. `外部咨询顾问`
   同时服务多个客户，需要快速归档并判断证据是否充分。

### 5.2 次级用户

1. `Internal reviewer`
   负责确认某个控制项下的证据是否成立。
2. `External auditor`
   主要消费输出结果，包括证据包、映射说明和补证状态。

## 6. 核心使用场景

### 场景 1：启动新的审计周期

用户创建一个审计 workspace，选择 `SOC 2` 模板，填写公司基础信息、审计周期、云环境、主要业务系统和团队角色。

### 场景 2：导入历史材料

用户上传制度文档、截图、CSV 资产清单，或连接 `Jira`、`Confluence` 拉取工单与知识库内容。系统对内容做解析、去重和结构化。

### 场景 3：自动映射控制项

系统根据控制项库和组织背景，对证据进行分类、摘要、时间识别，并映射到一个或多个控制项，同时展示证据理由与引用片段。

### 场景 4：发现缺口并补证

系统识别未覆盖控制项、证据过旧、证据不足或证据冲突，并输出优先级排序后的待补清单。

### 场景 5：生成人工可复核的审计包

系统按控制项生成说明草稿、证据引用列表、风险备注和待确认事项，reviewer 完成复核后导出审计包。

## 7. Jobs To Be Done

1. 当我要准备 `SOC 2` 审计时，我希望系统自动整理分散材料，这样我不需要手工从多个系统找证据。
2. 当我看到某个控制项时，我希望系统告诉我当前证据是否充分以及缺什么，这样我能优先补最关键的缺口。
3. 当我复核 AI 输出时，我希望看到清晰的引用来源和理由，这样我能快速判断它是否可信。
4. 当新一轮审计开始时，我希望复用历史证据和审计偏好，这样团队不会每次都从零开始。

## 8. v1 范围

### 8.1 范围内

1. 单租户工作区和单个组织 profile
2. 单一合规框架：`SOC 2`
3. 文件上传导入：`PDF`、`DOCX`、`PNG/JPG`、`CSV`
4. 外部连接：`Jira`、`Confluence`
5. 证据标准化、摘要、标签、时间识别、去重
6. 控制项映射与引用回链
7. 缺口识别、补证建议、复核工作台
8. 审计说明草稿与审计包导出
9. 组织级记忆与审计周期级记忆

### 8.2 范围外

1. 多框架交叉映射
2. 自动发送审计材料到第三方平台
3. 与云平台做深度实时配置检查
4. 移动端体验
5. 自动修复控制缺陷

## 9. 关键流程

### 9.1 端到端流程

1. 创建审计 workspace
2. 填写组织背景与审计周期信息
3. 导入文件或连接 `Jira` / `Confluence`
4. 系统解析证据，形成结构化 evidence item
5. `mapper agent` 将证据映射到控制项
6. `skeptic agent` 检查映射是否薄弱、过期或冲突
7. reviewer 在工作台中接受、修改或驳回结果
8. `writer agent` 生成审计说明与待补清单
9. 导出审计包

### 9.2 人工介入点

1. 初次导入后的组织背景确认
2. 映射结果复核
3. 风险等级与补证优先级确认
4. 审计包导出前最终确认

## 10. 功能需求

### FR-1 审计工作区与组织 Profile

系统必须支持创建组织和审计周期，并保存以下上下文：

1. 公司名称、行业、团队规模
2. 云环境与核心系统列表
3. 审计周期时间范围
4. 已知审计偏好与 reviewer 规则

### FR-2 控制项库

系统必须内置 `SOC 2` 控制项库，并支持：

1. 控制项说明与常见证据类型展示
2. 控制项状态追踪：未开始、部分覆盖、待复核、已确认、缺口
3. 每个控制项下的证据映射列表

### FR-3 证据导入与解析

系统必须支持以下导入方式：

1. 本地文件上传
2. `Jira` 工单导入
3. `Confluence` 页面导入

系统必须完成：

1. OCR 与文本抽取
2. 元数据提取：来源、创建时间、更新时间、作者、系统
3. 去重与近似重复合并
4. 证据类型分类：制度、截图、工单、资产清单、日志摘录

### FR-4 控制项映射

系统必须对每个 evidence item 输出：

1. 候选控制项列表
2. 映射置信度
3. 引用证据片段
4. 映射理由摘要

映射结果必须可追溯到原始内容，不允许只输出抽象结论。

### FR-5 缺口识别

系统必须识别以下缺口类型：

1. 控制项无证据
2. 控制项有证据但覆盖不足
3. 证据过期
4. 证据互相冲突
5. 证据与组织背景不一致

每个缺口必须附带补证建议、建议负责人和建议优先级。

### FR-6 复核工作台

reviewer 必须能够：

1. 接受 AI 映射
2. 修改控制项归属
3. 标记为证据不足
4. 输入反馈，作为后续 memory 训练材料
5. 查看该结论的所有引用来源

### FR-7 审计说明生成

系统必须为每个控制项生成说明草稿，包括：

1. 当前控制状态
2. 已采纳证据列表
3. 关键引用点
4. 未解决问题
5. 待补证建议

### FR-8 审计包导出

系统必须支持导出：

1. 控制项状态总览
2. 每个控制项下的证据与说明
3. 缺口清单
4. reviewer 决策记录

### FR-9 Memory

系统必须维护两类记忆：

1. `组织级记忆`
   包括常见系统、历史接受过的证据模式、审计偏好、控制解释习惯。
2. `任务级记忆`
   包括当前审计周期的 reviewer 决策、临时备注、补证进展和证据 freshness。

## 11. Agent 设计要求

v1 默认采用多 Agent 协作，不将所有能力塞进一个主 Agent。

1. `collector agent`
   负责证据采集、结构化、去重和初步分类。
2. `mapper agent`
   负责控制项候选匹配和理由生成。
3. `skeptic agent`
   负责发现证据不足、过期、冲突和错误映射。
4. `writer agent`
   负责生成审计说明、缺口总结和导出文本。
5. `review coordinator`
   负责在 LangGraph 中协调状态推进与人工复核节点。

## 12. LangGraph 状态流要求

v1 的编排必须是显式状态机，而不是自由对话链。建议最小状态集如下：

1. `workspace_setup`
2. `ingestion`
3. `normalization`
4. `mapping`
5. `challenge`
6. `human_review`
7. `package_generation`
8. `exported`

状态迁移要求：

1. `mapping` 之后必须经过 `challenge`
2. 存在中高风险缺口时必须进入 `human_review`
3. 任何导出动作前必须有 reviewer 确认

## 13. Skills / 工具集成要求

v1 工具能力以“读为主、写为辅”，避免高风险自动动作。

1. 文件上传与对象存储访问
2. `Jira` 查询与工单拉取
3. `Confluence` 页面拉取
4. 向量检索与全文检索
5. 审计包导出

禁止行为：

1. 未经确认自动修改源文档
2. 未经确认自动关闭工单或发送对外合规声明

## 14. 数据与对象模型

PRD 级别需固定以下核心对象：

1. `Organization`
2. `AuditCycle`
3. `Control`
4. `EvidenceItem`
5. `EvidenceMapping`
6. `Gap`
7. `ReviewerDecision`
8. `AuditNarrative`
9. `MemoryRecord`

关键字段要求：

1. 每个 `EvidenceItem` 必须带来源、时间、证据类型、原始链接或文件定位
2. 每个 `EvidenceMapping` 必须带置信度、理由、引用片段和 reviewer 状态
3. 每个 `Gap` 必须带严重度、建议动作和责任人

## 15. 评估、测试与优化要求

### 15.1 离线评估

需要构建一套标注数据集，至少覆盖：

1. 控制项与证据映射
2. 证据类型分类
3. 缺口识别
4. 说明文本忠实度

### 15.2 测试策略

1. 单元测试：解析、去重、引用定位、规则判断
2. 工作流测试：完整审计周期回放
3. 回归测试：对已通过 reviewer 的样本集做固定回放
4. 对抗测试：过期证据、冲突证据、伪相关证据

### 15.3 在线评估

1. reviewer 接受率
2. reviewer 修改类型分布
3. 每个控制项的人工耗时
4. 缺口关闭率

### 15.4 优化路径

v1 优先做：

1. prompt 与检索优化
2. 图状态与重试策略优化
3. 引用质量与 chunk 策略优化

微调仅用于：

1. 控制项分类
2. 证据类型识别
3. 补证优先级排序

v1 不微调主推理模型。

## 16. 非功能需求

1. `可追溯性`：每个 AI 结论都必须可回溯到原始证据
2. `隔离性`：组织间数据严格隔离
3. `可审计性`：保留 reviewer 操作日志与模型输出版本
4. `稳定性`：单次导入失败不应中断整个审计周期
5. `性能`：导入后首轮映射应在可接受时间内完成，v1 目标为中等规模审计在 `30` 分钟内产出首版结果

## 17. 风险与约束

1. 证据质量不稳定，OCR 和截图可读性会直接影响效果
2. 不同审计师对证据充分性的判断存在差异，需要通过 reviewer memory 渐进校准
3. 企业源数据格式混乱，会抬高解析成本
4. 如果没有引用回链，用户将很快失去信任

## 18. 商业化与包装假设

v1 适合的初始客户画像：

1. 正在准备首次 `SOC 2` 的 SaaS 创业公司
2. 缺少专职 GRC 团队的技术公司
3. 同时服务多家客户的安全咨询顾问

建议收费逻辑：

1. 按审计周期收费
2. 叠加按 workspace 或导出包数量收费
3. 高级集成与审计协作能力作为高阶套餐
